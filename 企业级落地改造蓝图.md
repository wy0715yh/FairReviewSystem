# 企业级落地改造蓝图（含 RAG、LangGraph、LangChain 升级）

## 目标定义
1. 从“单机可跑的 Flask 单体”升级为“多租户、可审计、可扩展、可观测”的企业级平台。
2. RAG 从“本地词频检索”升级为“可回溯、可评测、可运维”的标准检索链路。
3. 引入 `LangGraph` 做流程编排，`LangChain` 只做工具层（loader/splitter/retriever）能力。

## 当前到目标映射
1. `app.py` -> API Gateway + 审查服务 + 规则服务 + 知识库服务 + 报表服务。
2. `services/rag.py` -> RAG Service（向量索引、混合检索、重排、引用追踪）。
3. `services/audit_engine.py` -> Audit Orchestrator（LangGraph 主图）。
4. `services/storage.py` + json/txt -> PostgreSQL + Object Storage + Vector DB。
5. `templates/* + static/*` -> 企业前端（用户端 + 管理端，前后端分离）。

## 目标架构（企业版）
1. 前端层：`Web App`（审查台）+ `Admin Console`（租户/规则/知识库/权限）。
2. 接入层：`API Gateway`（鉴权、限流、审计、路由）。
3. 业务层：`Audit Service`、`RAG Service`、`Rule Service`、`Report Service`、`Identity Service`。
4. 编排层：`LangGraph Runtime`（节点状态、重试、人工复核分支、批量任务）。
5. 数据层：`PostgreSQL`（业务数据）+ `Redis`（缓存/队列状态）+ `Vector DB`（pgvector/Milvus/Chroma）+ `MinIO/S3`（文件）。
6. 运维层：`Prometheus + Grafana + Loki/ELK + OpenTelemetry`。

## 分阶段落地蓝图

### P0（2-4周）：稳态化与基础改造，最小可上线
1. 拆分代码结构为后端 API 包和前端包，保留单体部署但模块边界清晰。
2. 把 `audit_history.json/custom_rules.json/system_settings.json` 迁移到 PostgreSQL。
3. 增加用户体系最小版：登录、租户、RBAC（管理员/审查员/只读）。
4. LLM 网关化：统一超时、重试、熔断、错误码，不再在业务层散落调用。
5. 把当前兜底逻辑保留，并把“失败原因+请求链路ID”写入审计日志。
6. 增加 CI：lint + unit test + basic integration test。

交付物：
1. 可登录、多用户、多租户基础可用。
2. 核心审查链路稳定，错误可定位。
3. 数据不再落本地 JSON。

### P1（4-8周）：RAG 企业化 + LangGraph 引入
1. 建立 RAG 入库流水线：解析 -> 清洗 -> 切块 -> embedding -> 建索引 -> 版本标记。
2. 检索升级为混合检索：关键词/BM25 + 向量检索 + reranker。
3. 增加“引用追溯”强约束：每条结论必须带命中文档ID、片段ID、版本号。
4. 引入 `LangGraph` 编排审查图：
5. 节点A：输入预处理与敏感信息检测。
6. 节点B：RAG 检索与证据聚合。
7. 节点C：规则引擎匹配（企业规则 + 法规规则）。
8. 节点D：LLM 结构化输出（JSON Schema 校验）。
9. 节点E：风险分级与一致性检查。
10. 节点F：失败回退（重试/降级/人工复核队列）。
11. `LangChain` 使用范围限定为工具层：Document Loader、Text Splitter、Retriever 接口适配，不承担主流程控制。
12. 批量审查改为异步任务（队列 + 任务状态机 + 回调/轮询）。

交付物：
1. 可版本化知识库。
2. 可追溯的 RAG 证据链。
3. 可观测的 LangGraph 审查工作流。

### P2（8-12周）：企业治理与规模化
1. 微服务化部署（K8s），关键服务独立扩缩容。
2. 加入审批流和人工复核工作台（高风险自动转人工）。
3. 安全合规完善：字段加密、密钥托管、操作审计、防注入策略。
4. 多模型路由与成本治理：按租户配额、模型策略、token预算。
5. 建立评测体系：准确率、召回率、证据一致性、幻觉率、人工一致率。
6. 建立 SLA/SLO 与告警：P95 延迟、失败率、任务积压、索引失败率。

交付物：
1. 企业级可运营平台。
2. 成本、质量、合规三条线可管理。

## RAG 技术选型建议（企业视角）
1. PoC 或小规模可先用 Chroma。
2. 企业长期优先 `PostgreSQL + pgvector`（治理简单）或 `Milvus`（大规模向量能力更强）。
3. 不管选哪种库，必须补齐：版本管理、权限隔离、重建索引、回滚策略。

## LangGraph 与 LangChain 的使用边界
1. `LangGraph`：负责状态机、分支、重试、人工介入、长任务恢复。
2. `LangChain`：负责连接器和通用组件，不负责核心业务决策。
3. 法律规则判定、风险等级策略、报告模板建议保留自研，避免框架锁死。
